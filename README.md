# TSM for Warmane

针对 Warmane 3.3.5 WLK 服务器适配的 TradeSkillMaster 整合版。
完整汉化（含繁中）。

当前版本：**v2.9.9**

---

## 安装方法

1. 右上角点击绿色 **Code** 按钮 → **Download ZIP**
2. 解压后删除 `README.md`
3. 删除原先的 TSM 插件
4. 把新下载的插件文件剪切到魔兽插件目录（`Interface/AddOns/`）
5. 不建议直接覆盖

## 更新日志

**2026.06.10** — v2.9.9
- Crafting：紧急修复 v2.9.8 引入的「制造下一个」**一打开界面就是灰的、重登也无效**的问题。v2.9.8 用 `GetTradeskillRepeatCount()` 判断"客户端批量重复是否还在进行"来挡连点,实测该接口在 Warmane 3.3.5a 客户端上**空闲时也返回非零**,并不是"正在批量重复"的实时信号,于是按钮可用性判断和点击守卫恒被拦截,按钮永久锁死。现已撤掉对该接口的全部依赖:点击守卫改用插件自身的制作状态 `isCrafting`(完成/打断事件即时清除,看门狗 2 秒兜底清除,不存在锁死路径);看门狗判停阈值统一为 2 秒——正常批量两次施法之间的空档(约一个网络来回)远碰不到,被移动打断后事件名匹配得上的仍即时恢复,匹配不上的最多 2 秒自动恢复,卡灰不需要重登。v2.9.8 的核心修复——队列计数扣减与 `isCrafting` 解耦、任何情况下都扣得准——保留不变,防多做(越界)效果不受影响。

**2026.06.10** — v2.9.8
- Crafting：修复 v2.9.6 看门狗引入的**制作越界(多做)回归**。表现为材料备得刚刚好,批量制作中某个配方却多做了几个,把后面配方的墨水/羊皮纸吃掉,队列里剩余配方变红缺料;配合按键精灵连点「制造下一个」时尤其容易触发。根因有两层:①看门狗用「0.6 秒没在施法」判定制作已停止,但 Warmane 上批量制作两次施法之间的空档约等于网络延迟,卡顿时超过 0.6 秒就被误判,`isCrafting` 被中途清掉;②中文客户端的施法成功事件只能靠 `isCrafting` 兜底匹配本地化法术名来扣队列计数(`SpellName2ID` 是英文静态表),`isCrafting` 一被误清,批量里剩余的成功全部漏扣,队列残留虚高数字、按钮提前解锁,连点器再点一次就把同一配方又做了一遍。修复:①新增不被看门狗清掉的「最近制作法术」记录,成功事件用它兜底匹配,任何情况下队列计数都扣得准;②「制造下一个」的点击与按钮可用状态增加 `GetTradeskillRepeatCount()` 守卫,客户端批量重复没结束就一律不响应、不解锁,连点器点不出第二发 `DoTradeSkill`;③看门狗判停改为重复计数感知——repeat>0(批量还在进行)时要求连续 5 秒无施法才兜底恢复,正常批间空档碰不到,且恢复前会先撤销残存批量;repeat==0(移动打断后客户端已撤销批量)仍保持 0.6 秒快速恢复,v2.9.6 的按钮卡灰修复效果不变。

**2026.06.07** — v2.9.7
- 回退 v2.9.6 的「g20 扫描后自动还原原生排序」。该功能为了不拖慢"提前购买"(购买依赖那份降序扫描缓存按页快速定位)而做成**延迟**还原,但延迟期对边看边买的节奏不够友好:看货停顿一旦超过阈值,缓存被清、下一笔购买就会变慢。按使用偏好改回原状——g20 扫描后原生排序保持降序,需要时手动去原生 UI 点一下一口价列即可恢复升序。v2.9.6 的「制造下一个」按钮卡灰修复保留不变。

**2026.06.07** — v2.9.6
- Shopping：修复 `/gNN`(g20)扫描后**原生拍卖行的一口价排序被留在降序**(最贵的排最前),导致之后日常(非 g20)随手搜东西也跟着"最贵的先显示"、得手动去原生 UI 再点一次一口价列才恢复的问题。现在 g20 扫描会在玩家停止操作几秒后,自动把原生排序还原成升序(等于普通搜索结束后的状态)。还原特意做成**延迟**、而不是扫完立刻执行:很多人不等扫完就点结果提前购买,而提前购买依赖那份"降序"扫描缓存按页快速定位;若马上翻成升序会让缓存页码失配、购买反而变慢。因此提前购买期间保持降序+缓存(购买照样快),每点一次购买都会把还原往后推,等购买停下 3 秒才还原。g20 扫描期间仍按降序把大堆叠抢到前几页(加速功能不变);只有 g20 才触发还原,日常使用与加该功能之前一致。
- Crafting：修复"制造下一个"被**移动打断**后按钮一直灰着、得小退重登才能恢复的问题。被移动打断的制作不产出成品(没有 `TRADE_SKILL_UPDATE` 刷新),且打断事件名在中文客户端不一定能匹配上,旧逻辑下 `isCrafting` 状态卡住、按钮再不解锁。新增制作看门狗:制作期间每 0.2s 轮询一次 `UnitCastingInfo`,一旦制作真正停下(打断 / 失败 / 没起手)就兜底清掉状态并刷新队列,约 0.6 秒内自动恢复按钮,无需重登。批量制作两次之间的无缝衔接不会被误判为"已停下"。

**2026.06.04** — v2.9.5
- AuctionScanning：修复**每次扫描"第一个物品"偶发以普通价发布**的冷启动竞态。压价扫描刚开始时 AH 浏览列表往往是空的，而设排序触发的 `AUCTION_ITEM_LIST_UPDATE` 回声可能赶在服务器返回真正结果之前先到，此时拿到 `total==0`，旧逻辑直接当成"0 页 → 扫描完成"，于是本轮最先处理的那个物品（例如 主人的召唤雕文）被误判成"AH 上无人挂" → 走 `postingNormal`，既不压价也读不到自己已有的挂单。现在对查询型扫描：第 0 页若 `total==0` 且查询其实还在途（`CanSendAuctionQuery` 仍为否），就有界软重试等真结果到货；服务器确实回 0 结果时立即收尾，不拖慢真·无人挂的物品。

**2026.06.04** — v2.9.4
- AuctionScanning：修复拍卖扫描偶发以**降序**发出查询，导致 fastScan 只读到前几页最贵的几条、漏掉真正最低价的问题。表现为 附魔武器-狂暴 / 雕文等明明 AH 上有 677g 的低价单，插件却抓到 888g+ 当"最低价"，于是走「高于最高价，以普通价发布」或「以正常价格发布」，按普通价直接挂、也读不到自己已挂的单。根因是 `/gNN` 反向扫描上线后排序方向改成依赖 `IsAuctionSortReversed()` 判断要不要再翻一次，而 Warmane 3.3.5a 上这个查询常返回"翻转前"的旧值，把升序又翻回降序（是否触发取决于上一次扫描遗留的排序状态，所以偶发）。现改为 `SortAuctionClearSort` + 确定性升序，全程不再读 `IsAuctionSortReversed`，只有 `/gNN` 最小堆叠模式才切降序。

**2026.06.03**
- Shopping：新增最小堆叠过滤 `/gNN`（例：`无限之尘/exact/g20` 只显示 ≥20 的堆叠）。带该过滤时扫描改为按总价**降序**，大堆叠落到最前几页，几秒内就在增量结果里出现，无需等整轮扫描结束；结果表仍按单价排序，最便宜的合格堆叠在最上面。
- AuctionControl：一口价失败（索引过期 / 弹"未找到指定物品"）时**自动重新查询并把确认框恢复成"可购买"**（最多 3 次后才放弃），复刻手动"再搜一次就能买"的恢复动作 —— 不再卡在"寻找物品…"、也不用关掉重搜整张 AH，再点一下即可买到（出价本身仍需点击，是 3.3.5a 的限制）。
